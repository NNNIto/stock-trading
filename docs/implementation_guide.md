## 0. このドキュメントの使い方

### 0.1 前提
- 本手順書は requirements.md / scenarios.md / architecture.md を前提とする
- Claude Codeに作業を依頼する際、関連文書をコンテキストに含めること
- 1タスク=1コミット（Conventional Commits形式）を原則とする
- 各タスク完了時にテストを実行し、グリーンを確認してから次へ進む

### 0.2 Claude Codeへの基本指示テンプレート

各タスク依頼時、以下を冒頭に付与すると精度が上がる。

```
あなたは個人の株式投資自動化システムを実装します。
前提文書（requirements.md, scenarios.md, architecture.md）を必ず参照してください。

制約:
- Python 3.11+、uv、Polars（メイン）、DuckDB、pandas-ta を使用
- 過学習回避が最優先。バックテストエンジンは挙動が完全に理解できる自作実装
- 設定はYAML外出し、ロジックはコード
- 全公開関数に型ヒント、重要モジュールはmypy通過
- 1タスク完了ごとにテストを書き、pytestグリーンを確認
- コミットはConventional Commits形式

このタスクのゴール: <タスク内容>
完了条件: <受け入れ基準>
```

### 0.3 進捗管理
- 各タスクのチェックボックスを完了時にチェック
- ブロッカーが出たら該当タスクに「⚠️ブロッカー: 内容」を追記

完了条件: バックテストとライブが同一実装なら完全一致が確認でき、
意図的にライブ側ロジックを変えたケースで不一致が検出される

## 付録C: 注意事項

- 各タスクは前提文書（requirements.md, scenarios.md, architecture.md）を必ず参照
- パラメータのハードコード禁止（必ずYAML経由）
- バックテスト改善時、必ずアウトオブサンプルでも検証（過学習チェック）
- シナリオ変更は必ずGitコミット（変更理由・改善幅をメッセージに記載）
- yfinance障害時は代替データソース（J-Quants/Stooq/Alpha Vantage）へ切替検討
- Phase 5以降は「開発」ではなく「運用」フェーズ。焦って実弾投入せず基準を守る
- 品質保証4レイヤ（architecture.md 7-A）: L1静的ゲートは全コミット必須、
  L3金融ロジック検証はバックテスト前に必須通過、L4健全性検証はバックテスト
  後に自動実行、L2コードレビューはPhase 3以降の重要モジュールで実施
- L2/L3の軽量運用（エージェントレビュー）は必ず実装セッションとは別セッション
  で行う。自己レビューは盲点を見逃す
